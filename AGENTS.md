# AGENTS.md

## 기본 원칙

- 기존 기능을 임의로 삭제하지 않는다.
- 사용자가 요청한 범위만 수정한다.
- 대규모 구조 변경은 사전에 이유를 설명한다.
- 해운회사 업무용 시스템에 맞는 안정적이고 전문적인 UI를 유지한다. 현재 프로젝트는 운세/라이프스타일 앱이므로 이 원칙의 적용 여부는 확인 필요.
- 이 프로젝트는 운세/라이프스타일 모바일 웹앱이므로 밝고 신뢰감 있는 UI를 유지한다.
- 초보자도 유지보수할 수 있도록 가독성 있는 코드를 작성한다.

## 코드 수정 원칙

- React 컴포넌트, 유틸, 데이터 파일 분리 구조를 최대한 유지한다.
- 기존 함수명, 저장 키, 컴포넌트 props, class명을 임의로 변경하지 않는다.
- 기존 localStorage 데이터 구조와 호환되도록 수정한다.
- 수정 범위를 최소화한다.
- 불필요한 라이브러리를 추가하지 않는다.
- 기능 파일을 수정하기 전 관련 파일을 먼저 읽고 현재 흐름을 이해한다.

## 보안 원칙

- API Key, DB URL, Secret Key를 코드에 직접 작성하지 않는다.
- 환경변수는 `.env` 또는 Vercel Environment Variables로 관리한다.
- `.env` 파일은 Git에 커밋하지 않는다.
- `.env.example`에는 실제 값이 아닌 변수명만 작성한다.

## Git 작업 원칙

- `main` 브랜치에 직접 기능 수정하지 않는다.
- 기능별 브랜치를 생성한다.
- 브랜치 이름 예시:
  - `feature/login-ui`
  - `feature/ai-consult-api`
  - `fix/vercel-api-error`
  - `chore/project-workflow-setup`
- 작업 후 PR을 생성한다.
- PR 생성 전 테스트 방법을 문서화한다.
- 작업 중 미커밋 변경사항이 있으면 덮어쓰지 말고 먼저 사용자에게 알린다.

## Vercel 배포 주의사항

- `main` 브랜치 merge 후 자동 배포될 수 있으므로 `main` push 전 반드시 검토한다.
- 환경변수 누락 여부를 확인한다.
- API route 경로 문제를 확인한다.
- 정적 파일 경로 문제를 확인한다.
- Vercel Preview에서 화면과 콘솔 오류를 확인한다.

## 작업 완료 시 반드시 보고할 내용

Codex는 작업 완료 후 아래 내용을 요약해야 한다.

1. 생성/수정한 파일 목록
2. 각 파일의 목적
3. 기존 기능 파일을 수정했는지 여부
4. 테스트한 명령어
5. 남은 확인사항
6. ChatGPT에 검토 요청할 요약문
