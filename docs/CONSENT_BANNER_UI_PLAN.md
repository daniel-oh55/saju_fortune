# CONSENT_BANNER_UI_PLAN

## consent storage 유틸 준비 상태

- `src/utils/consentPreferencesStorage.js`에서 consent preferences 저장 유틸을 제공합니다.
- localStorage key는 `harupuli_consent_preferences_v1`입니다.
- 현재 PR에서는 storage 유틸만 준비하며, 실제 동의 배너 UI에는 아직 연결하지 않습니다.
- 실제 동의 배너 UI 구현 PR에서 이 유틸을 사용해 동의 상태를 저장하고 조회할 수 있습니다.
- consent storage 유틸은 준비되었지만 production UI에는 아직 연결하지 않습니다.

이 문서는 하루풀이 MVP 기준의 동의 배너 UI 구현 전 설계 문서입니다.
실제 광고 SDK, 분석 SDK, 로그인, 결제 기능 도입 시 최신 법령과 각 SDK provider 정책을 기준으로 별도 검토가 필요합니다.
이번 문서는 구현 계획이며, 실제 동의 배너 UI와 consent localStorage key는 아직 구현하지 않습니다.

## 1. 목적

- 향후 실제 광고 SDK 또는 분석 SDK 도입 시 필요한 동의 배너 UI 구조를 사전에 정리합니다.
- 사용자가 어떤 데이터 사용에 동의하는지 쉽게 이해할 수 있게 합니다.
- 동의하지 않아도 기본 운세 기능을 사용할 수 있는 UX를 유지합니다.
- 광고 해금과 개인정보 동의를 혼동하지 않도록 합니다.

## 2. 현재 MVP 상태

- 실제 광고 SDK 없음
- 실제 분석 SDK 없음
- 실제 결제 SDK 없음
- 실제 동의 배너 UI 없음
- mock rewarded ad provider만 사용
- `localStorage` 중심으로 동작
- `PrivacyInfoPage`에서 현재 MVP 기준 개인정보 안내 제공

## 3. 동의 배너 표시 조건 초안

향후 실제 SDK 도입 시 표시 조건 후보:

- 실제 광고 SDK가 활성화된 경우
- 실제 분석 SDK가 활성화된 경우
- consent preferences가 저장되어 있지 않은 경우
- privacy/consent policy version이 변경된 경우
- 사용자가 설정에서 동의 상태를 초기화한 경우

현재 MVP:

- 위 조건은 문서상 후보이며 실제 코드에 구현하지 않습니다.

## 4. 배너 기본 문구 초안

제목:

하루풀이의 데이터 사용 안내

본문:

하루풀이는 기본 운세 제공을 위해 브라우저 저장소를 사용합니다. 향후 광고나 분석 기능을 사용하는 경우, 관련 데이터 사용에 대한 선택 동의를 받을 예정입니다.

버튼:

- 동의
- 설정
- 나중에
- 거부

주의:

- “동의하지 않으면 운세를 볼 수 없습니다” 같은 과도한 표현을 사용하지 않습니다.
- 결제처럼 보이는 표현을 사용하지 않습니다.
- 광고 해금과 개인정보 동의를 같은 의미처럼 표현하지 않습니다.

## 5. 설정 모달 또는 설정 페이지 초안

향후 설정 화면에서 관리할 항목:

- 필수 저장 안내
- 분석 데이터 사용 동의
- 광고 데이터 사용 동의
- 맞춤형 광고 동의
- 동의 철회
- 개인정보 안내 보기
- 저장 데이터 삭제

현재 MVP:

- 실제 설정 UI는 구현하지 않습니다.
- 문서상 구조만 정리합니다.

## 6. 동의 상태 후보 구조

향후 localStorage key 후보:

`harupuli_consent_preferences_v1`

예상 구조:

```json
{
  "analytics": false,
  "ads": false,
  "personalizedAds": false,
  "policyVersion": 1,
  "updatedAt": "2026-06-10T00:00:00.000Z"
}
```

주의:

- 이번 PR에서 실제 key를 코드에 추가하지 않습니다.
- 실제 구현 시 별도 PR에서 consent storage 유틸을 만듭니다.
- 기존 localStorage key와 충돌하지 않게 합니다.

## 7. 동의 상태 흐름

1. 첫 방문
   - consent preference 없음
   - 배너 표시 후보

2. 사용자가 동의
   - analytics/ads/personalizedAds 상태 저장 후보
   - 필요한 SDK 호출 허용 후보

3. 사용자가 거부
   - 선택 SDK 호출 차단 후보
   - 기본 운세 기능은 계속 사용 가능

4. 사용자가 설정 선택
   - 상세 설정 화면 또는 모달 표시 후보

5. 사용자가 나중에 선택
   - 일정 기간 뒤 다시 표시할지 별도 검토
   - 과도하게 반복 표시하지 않음

6. 정책 버전 변경
   - 기존 동의 상태가 있어도 재동의 필요 여부 검토

## 8. 광고 해금 UX와의 관계

- 광고 해금은 선택형 심화 콘텐츠 접근 방식입니다.
- 개인정보 동의는 광고 SDK 또는 분석 SDK의 데이터 사용 동의입니다.
- 두 개념이 UI에서 혼동되지 않도록 문구를 분리합니다.
- 광고 동의가 필요한 실제 광고 SDK를 도입할 경우, 광고 해금 버튼 클릭 시 동의 상태를 확인할 수 있습니다.
- 동의가 없어 실제 광고를 표시할 수 없는 경우에도 기본 운세는 계속 제공합니다.

## 9. 거부/철회 UX 원칙

- 거부해도 기본 운세 기능은 유지합니다.
- 동의를 철회하면 관련 SDK 호출을 중지합니다.
- 철회 후 저장 데이터 삭제는 별도 데이터 초기화 기능과 연결할 수 있습니다.
- 거부와 철회 문구는 짧고 차분하게 작성합니다.
- 불안감이나 손실 회피를 이용하지 않습니다.

## 10. 금지 문구

- 동의하지 않으면 운세를 볼 수 없습니다.
- 지금 동의하지 않으면 불리합니다.
- 동의하지 않으면 광고 보상을 받을 수 없습니다.
- 결제해야 계속 이용할 수 있습니다.
- 거부하면 일이 꼬일지 모릅니다.

## 11. 구현 전 체크리스트

- 실제 SDK provider 확정
- 광고/분석 SDK가 쿠키 또는 광고 식별자를 사용하는지 확인
- 개인정보 처리방침 업데이트
- 동의 배너 문구 검토
- 동의 상태 저장 유틸 설계
- 설정 화면 내 동의 관리 위치 확정
- 거부 후 기본 기능 유지 확인
- 미성년자/민감정보 관련 정책 확인
- profile 원본 정보를 SDK로 보내지 않는 구조 확인

- 동의 배너 UI 추가
- 동의 설정 모달 추가
- SettingsPage 동의 관리 섹션 추가
- 실제 광고 SDK adapter 추가
- 실제 분석 SDK 도입 여부 검토
