# COOKIE_AD_CONSENT_UX

## 개인정보 처리방침 URL 연계

- 광고/분석/개인화 동의 UX는 개인정보 처리방침 URL과 함께 안내되어야 합니다.
- 현재 실제 광고 SDK는 미연동 상태입니다.
- 실제 광고 SDK 연결 전 개인정보 처리방침과 동의 UX를 함께 갱신해야 합니다.

## placement와 consent의 관계

- placement ID는 광고 위치 식별자이고 ads consent는 실제 SDK 호출 허용 조건입니다.
- ads consent가 false이면 실제 SDK provider 호출을 하지 않습니다.
- placement ID가 설정되어 있어도 consent 조건을 먼저 확인합니다.
- 현재 MVP에서는 mock provider를 유지하며 실제 광고 SDK 호출은 없습니다.

## ads consent gate 적용 상태

- ads consent가 실제 SDK 호출 조건으로 연결되는 scaffold가 추가되었습니다.
- ads 미동의 상태에서도 기본 운세와 무료 해석은 계속 이용할 수 있습니다.
- 현재 MVP에서는 mock provider가 기본값이며 실제 광고 SDK 호출은 없습니다.

## SDK adapter와 동의 상태

- SDK adapter scaffold는 준비되었지만 실제 광고 SDK는 아직 호출하지 않습니다.
- 실제 SDK 호출 전 ads consent 확인 로직은 별도 PR에서 연결합니다.
- 현재 MVP에서는 consent 상태 저장과 광고 SDK 호출이 직접 연결되어 있지 않습니다.

## rewarded ad SDK 연동과 동의 상태

- 실제 rewarded ad SDK를 호출하기 전 ads consent 상태를 확인하는 방향을 검토한다.
- ads consent가 false인 경우 실제 광고 호출 대신 데이터 사용 설정으로 안내할 수 있다.
- 실제 구현 전 `docs/REWARDED_AD_SDK_INTEGRATION_PLAN.md`를 참고한다.
- 현재 MVP에서는 mock provider만 사용한다.

## consent storage 유틸

- 동의 상태 저장 후보 구조는 `src/utils/consentPreferencesStorage.js` 유틸로 준비합니다.
- 현재 MVP에서는 동의 배너 UI를 표시하고, 선택 상태를 localStorage에 저장합니다.
- 이 배너는 실제 SDK 동의가 아니라 향후 기능 대비 동의 상태 관리입니다.
- 실제 광고/분석 SDK 도입 시 문구와 동의 항목을 다시 검토해야 합니다.

## 동의 배너 UI 구현 계획 문서

- 실제 동의 배너 UI를 구현하기 전 `docs/CONSENT_BANNER_UI_PLAN.md`를 참고합니다.
- 해당 문서는 배너 문구, 버튼 구성, 동의 상태 후보 구조, 광고 해금 UX와의 관계를 정리합니다.
- 현재 MVP에서는 실제 동의 배너 UI와 consent localStorage key를 구현하지 않습니다.

이 문서는 하루풀이 MVP 기준의 쿠키/광고 동의 UX 검토 문서입니다.
법률 검토를 완료한 최종 정책이 아니며, 실제 광고 SDK, 분석 SDK, 로그인, 결제 기능 도입 시 최신 법령과 각 SDK provider 정책을 기준으로 별도 검토가 필요합니다.

## 1. 목적

- 실제 광고 SDK 또는 분석 SDK 도입 시 사용자 동의 UX를 어떻게 설계할지 정리합니다.
- 사용자가 어떤 데이터 사용에 동의하는지 명확히 이해할 수 있게 합니다.
- 광고와 분석 기능을 도입하더라도 기본 운세 이용을 과도하게 제한하지 않도록 합니다.
- 동의 UX가 불안감, 결제 압박, 광고 시청 강요처럼 보이지 않게 설계합니다.

## 2. 현재 MVP 상태

- 실제 광고 SDK는 연결되어 있지 않습니다.
- 실제 분석 SDK는 연결되어 있지 않습니다.
- 실제 결제 SDK는 연결되어 있지 않습니다.
- 보상형 광고는 mock provider만 사용합니다.
- 서버/DB 저장 없이 `localStorage` 중심으로 동작합니다.
- profile, fortune cache, reward unlock, saved readings, visit streak 등이 브라우저 저장소에 저장될 수 있습니다.
- 이 문서는 실제 SDK 도입 전 준비 문서입니다.

## 3. 동의 UX 검토가 필요한 경우

- 실제 광고 SDK 도입
- 맞춤형 광고 사용
- 광고 식별자 또는 쿠키 사용
- 분석 SDK 도입
- 로그인 기능 도입
- 서버/DB 저장 도입
- 결제 기능 도입
- 외부 provider로 데이터 전송이 발생하는 기능 도입

## 4. 아직 구현하지 않는 이유

현재 MVP 기준:

- 실제 광고/분석 SDK가 아직 없기 때문에 동의 배너 UI를 구현하지 않습니다.
- 먼저 개인정보 처리방침 초안과 데이터 맵을 정리합니다.
- 실제 SDK 도입 시 provider 정책과 법률 검토를 바탕으로 별도 PR에서 동의 UX를 구현합니다.

## 5. 동의 배너 UX 원칙

- 사용자가 이해하기 쉬운 문구를 사용합니다.
- `동의`, `설정`, `거부` 선택지를 명확히 제공합니다.
- 거부해도 기본 운세 기능은 사용할 수 있도록 설계합니다.
- 맞춤형 광고 동의와 필수 저장 기능을 구분합니다.
- 동의하지 않으면 광고 해금이 제한될 수 있는 경우 차분하게 안내합니다.
- 불안감이나 손실 회피를 이용해 동의를 유도하지 않습니다.
- `지금 동의하지 않으면 일이 꼬입니다` 같은 표현은 금지합니다.
- 결제처럼 보이는 표현은 사용하지 않습니다.
- 광고 해금은 선택형 심화 콘텐츠로 유지합니다.

### 선택 광고

현재 MVP:

- 실제 광고 SDK 없음

### 선택 마케팅

현재 MVP:

## 7. 동의 저장 구조 초안

아래 구조는 향후 구현 후보이며, 이번 PR에서 실제 코드에 추가하지 않습니다.

향후 localStorage key 후보:

`harupuli_consent_preferences_v1`

예상 저장 구조:

```json

  "analytics": false,
  "ads": false,
  "personalizedAds": false,
  "updatedAt": "2026-06-10T00:00:00.000Z",
  "version": 1
}
```

주의:

- 이번 PR에서는 위 key를 실제 코드에 추가하지 않습니다.
- 문서상 후보로만 기록합니다.
- 실제 SDK 도입 시 별도 PR에서 구현합니다.

## 9. 광고 SDK 도입 전 확인 항목

- provider가 웹앱에서 사용 가능한지
- 쿠키 사용 여부
- 광고 식별자 사용 여부
- 맞춤형 광고 사용 여부
- 사용자 동의 API 제공 여부
- 동의 철회 방법
- 미성년자 대상 제한 여부
- 광고 provider로 profile 원본 정보가 전달되지 않는지
- 생년월일, 출생시간, 성별, 사주 정보가 광고 provider로 전달되지 않는지
- 개인정보 처리방침 업데이트 필요 여부
- 별도 동의 배너 필요 여부

## 10. 분석 SDK 도입 전 확인 항목

- 수집 이벤트 목록 정의
- 수집하지 않을 정보 정의
- profile 원본 정보 수집 금지
- 생년월일, 출생시간, 성별 직접 전송 금지
- IP, 기기 정보, 브라우저 정보 처리 여부
- 사용자 동의 필요 여부
- 개인정보 처리방침 업데이트 필요 여부

## 11. 거부/철회 UX

- 동의를 거부해도 기본 운세 이용은 가능해야 합니다.
- 광고 SDK 동의가 필요한 광고 기능은 제한될 수 있습니다.
- 제한되는 경우에도 차분하고 짧게 안내합니다.
- 동의 철회 후에는 관련 SDK 호출을 중지해야 합니다.
- 이미 저장된 localStorage 데이터 삭제는 별도 데이터 초기화 기능과 연결할 수 있습니다.

- 실제 동의 배너 또는 동의 설정 화면은 아직 구현하지 않습니다.
- 실제 광고/분석 SDK 도입 시 `PrivacyInfoPage`와 동의 배너/설정 화면의 문구를 함께 검토합니다.
