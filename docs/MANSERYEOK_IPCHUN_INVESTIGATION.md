# MANSERYEOK_IPCHUN_INVESTIGATION

## 목적

1990년 입춘 절입 시각과 사주 년주/월주 전환 기준을 검증하기 위한 조사 문서다.
현재 목적은 계산 엔진을 수정하는 것이 아니라, 외부 기준값 간 차이를 기록하고 향후 엔진 보정 여부를 판단하는 것이다.

## 현재 검증 대상

### solar_ipchun_boundary

- 입력: 1990-02-04 10:30
- calendarType: solar
- gender: male
- location: Seoul, South Korea
- 현재 expected: 기사년 정축월 경자일 신사시
- 현재 engine actual: 경오년 무인월 경자일 신사시
- mismatchFields: pillars.year, pillars.month

## 현재 기준별 결과

### sky.told.me

- 결과: 기사년 정축월 경자일 신사시
- 기준 메모: 출생도시 Seoul, South Korea 기준
- 태양시 보정 여부: 확인 필요
- 입춘 절입 시각 표시 여부: 확인 필요

### 포스텔러

- 결과: 기사년 정축월 경자일 신사시
- 기준 메모: 사용자 제공 이미지 기준
- 태양시 보정 여부: 확인 필요
- 입춘 절입 시각 표시 여부: 확인 필요

### lunar-javascript exact API

- 결과: 경오년 무인월 경자일 신사시
- 사용 API:
  - getYearInGanZhiExact
  - getMonthInGanZhiExact
- 기준 메모: 라이브러리 내부 절기 기준
- 태양시 보정 여부: 미적용

## 확인해야 할 핵심 질문

1. 1990년 입춘 절입 시각은 한국 표준시 기준 몇 시인가?
2. 1990-02-04 10:30 Seoul은 입춘 전인가, 입춘 후인가?
3. sky.told.me와 포스텔러는 태양시 보정을 적용하는가?
4. lunar-javascript exact API의 절기 기준은 한국 표준시 기준과 동일한가?
5. 사주 년주/월주 전환 기준을 한국 서비스 기준으로 맞출 것인가, 라이브러리 기준으로 유지할 것인가?

## 판단 기준 초안

- 공신력 있는 절기 시각 자료에서 1990-02-04 10:30이 입춘 전으로 확인되면:
  - sky.told.me / 포스텔러 기준을 우선 검토한다.
  - 엔진의 년주/월주 보정 필요성이 높아진다.

- 공신력 있는 절기 시각 자료에서 1990-02-04 10:30이 입춘 후로 확인되면:
  - lunar-javascript exact API 기준을 우선 검토한다.
  - 기존 expected의 기준값 재검토가 필요하다.

- 출생도시 기반 태양시 보정에 따라 입춘 전후 판단이 달라진다면:
  - v0 엔진에서는 태양시 보정을 적용하지 않는다는 현재 정책을 유지한다.
  - 해당 샘플은 reference_conflict 또는 policy_pending 후보로 분류한다.

## 현재 결론

아직 최종 결론을 내리지 않는다.
solar_ipchun_boundary는 엔진 보정 후보이지만, 실제 엔진 수정 전 1990년 입춘 절입 시각과 외부 서비스의 태양시 보정 여부를 추가 확인한다.
