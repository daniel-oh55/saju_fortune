# SAVED_READINGS

## 목적

사용자가 마음에 드는 풀이를 저장해 나중에 다시 볼 수 있도록 한다.
현재는 서버 없이 localStorage 기반으로 시작하며, 기존 운세 계산 로직과 광고 해금 구조에는 영향을 주지 않는다.

## localStorage key

- `harupuli_saved_readings_v1`

## 저장 구조

```json
{
  "items": [
    {
      "id": "fortune:2026-06-10:overall",
      "type": "fortuneCategory",
      "title": "오늘운세",
      "summary": "오늘의 핵심 요약",
      "body": "저장된 풀이 본문",
      "tags": ["정리", "균형"],
      "dateKey": "2026-06-10",
      "savedAt": "2026-06-10T05:00:00.000Z"
    }
  ]
}
```

## 저장 가능한 항목

- 오늘운세 상세 요약
- 사주 흐름 무료 영역 요약

## 저장하지 않는 항목

- 생년월일
- 출생시간
- 성별
- 음력/양력 원본 설정
- 23시 이후 자시 정책
- 원본 profile 객체

## 중복 저장 처리

- 같은 `id`를 다시 저장하면 중복 항목을 만들지 않는다.
- 기존 항목의 내용과 `savedAt`을 최신 값으로 업데이트한다.
- 목록은 최근 저장순으로 정렬한다.

## 최대 저장 개수

- 최대 50개까지 유지한다.
- 50개를 넘으면 최근 저장 항목 중심으로 보관한다.

## localStorage 기반으로 시작하는 이유

- 로그인 없이 MVP에서 바로 사용할 수 있다.
- 저장 UX를 빠르게 검증할 수 있다.
- 서버 비용과 개인정보 처리 범위를 늘리지 않고 시작할 수 있다.

## 향후 확장

- 로그인 기능이 생기면 DB 저장으로 확장할 수 있다.
- 여러 기기 간 저장 항목 동기화는 Firebase/Supabase/Neon 같은 저장소 도입 후 검토한다.
- 서버 저장을 도입해도 기존 localStorage key는 마이그레이션 대상으로만 다룬다.

## 개인정보 주의사항

- 저장 항목에는 생년월일, 출생시간, 성별 같은 원본 프로필 정보를 넣지 않는다.
- 저장된 텍스트는 사용자가 화면에서 확인한 풀이 요약과 본문 중심으로 구성한다.
- 민감할 수 있는 원본 사주 입력값은 저장하지 않는 것을 원칙으로 한다.
